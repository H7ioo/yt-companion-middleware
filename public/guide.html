01 How it works

One broadcast target at a time, edited safely, mirrored into a cache that Companion can poll for free.

Two states, one target

The app always points at a single broadcast and figures out which one on every trigger:

  • Idle — no live stream. Edits land on your channel's persistent broadcast container (the default "upcoming" broadcast).
  • Live — a stream is active. Edits land on the currently active broadcast. If more than one is active, the app picks the most recent actualStartTime and logs a warning.

Brand-new channel with no container at all? The action fails cleanly with NO_TARGET_FOUND rather than inventing one.

Read-modify-write, always. The app never blind-overwrites metadata. Every trigger does GET → merge your changes over the live object → PUT the whole thing back. Fields it doesn't manage — thumbnail, game ID, everything else — pass through untouched, so nothing you set in YouTube Studio gets wiped mid-stream.

What a trigger resolves

  1. title, description, privacyStatus — from the preset or ad-hoc payload.
  2. category — preset override → else app default → else left as-is.
  3. streamBoundId (stream key binding) — same order: preset override → app default → as-is.
  4. Everything else — copied straight from the live object.

The state cache

Every feedback endpoint is served from an in-memory cache, never a live YouTube call. The cache refreshes after every successful action and on a background timer (REFRESH_INTERVAL_SECONDS, default 60s) to catch out-of-band changes like a stream you ended from YouTube Studio. Companion can poll every 5 seconds and spend zero YouTube quota.

02 Install & run

Node 20+, one OAuth refresh token, a JSON file on disk. That's the whole footprint.

Configure the environment

Copy .env.example to .env and fill it in. The YouTube refresh token is obtained once through Google's OAuth consent flow and stays server-side — it is never exposed to Companion or any browser. 

# YouTube OAuth 2.0 (Google Cloud Console → OAuth client)
YT_CLIENT_ID=your-client-id.apps.googleusercontent.com
YT_CLIENT_SECRET=your-client-secret
YT_REFRESH_TOKEN=your-refresh-token

PORT=8080
DATA_DIR=./data            # JSON store; mount as a Docker volume
REFRESH_INTERVAL_SECONDS=60
HEALTH_FAILURE_THRESHOLD=3   # fails in a row before health escalates
YT_REGION_CODE=US            # drives the assignable category list
YT_QUOTA_LIMIT=10000          # daily quota budget, shown on the dashboard

Run it

# development — API on :8080 (tsx watch, auto-restart)
npm install
npm run dev

# hot-reloading dashboard (separate terminal, Vite dev server)
npm --prefix web run dev

# production — build the SPA + server, then serve both from :8080
npm run build:all
npm start

npm run dev runs the API only. In development the dashboard is served by Vite on its own port (proxying /api and /guide back to :8080); the production build bundles the SPA so npm start serves everything from a single port.

Open http://<host>:8080 for the dashboard, /docs for the interactive API console, and this page at /guide. The API lives under /api.

Storage. Presets, default settings, the webhook URL, and cached state all live in a single JSON file under DATA_DIR. Writes are atomic (temp file, then rename) so a crash mid-write can't corrupt it. Mount that directory as a Docker volume and your config survives redeploys.

03 No login (LAN-trust)

There is no token, no header, no login screen. Anything on your LAN can trigger the app — and that is the intended design.

The threat model is a personal tool on a private network: the worry is a random device on the same LAN casually hitting an endpoint, not a determined attacker. If someone already has access to this host, API auth wouldn't have saved you anyway. So the Bearer token from earlier versions is gone — no Authorization header to set in Companion, nothing to regenerate.

Do not port-forward this to the internet. Every action and feedback endpoint is unauthenticated. Keep it on the LAN (or behind a VPN) and you're within the model; expose it publicly and anyone can rewrite your broadcast.

Because Companion buttons may have been wired to either path historically, /api/action/* and /api/dashboard/action/* are the same handler. Use whichever you like; both work.

04 The web interface

Everything you configure lives in the dashboard. Companion only ever fires what you set up here.

Presets

A preset is a named bundle of broadcast metadata you can fire in one press. Create / edit / duplicate / delete from the grid. Each preset carries:

  • Title, description, privacy — always set.
  • Title / description fallback — optional whole-sentence alternates used when a field has an unresolved variable (see §05). Leave blank to make the variable required.
  • Category and stream binding — optional overrides. Leave blank and the preset shows "inherits default"; the app-level default fills in at trigger time.

As you type a {variable} into the title or description, a chip appears under that field naming the variable (and its inline default, if any) — immediate confirmation that it's now a fill-in the popup will prompt for.

Every preset displays its id for copy-paste into a Companion button payload. Duplicate clones a preset (title suffixed "(copy)") so you can spin a variant without retyping. Delete one and any Companion button still pointing at it simply gets INVALID_PRESET on its next press — no cascade, no cleanup.

Backup & restore

Export downloads every preset as a JSON file; Import reads one back. Import offers replace (restore a backup, keeping IDs) or merge (append the file's presets as copies with fresh IDs) — handy for cloning a set between hosts.

App-level defaults

The Settings panel sets defaultCategory and defaultStreamBoundId. These are the baseline used whenever a preset or ad-hoc update doesn't override them.

Ad-hoc update

Need to push a one-off change without making a preset? The ad-hoc modal takes title, description, privacy, and optional category / stream-binding overrides, and shows a badge telling you whether the change will hit the idle container or a live broadcast.

State rail

A live readout of what the app currently sees: on-air state, the on-screen title, health, whether the runner is busy, and remaining YouTube quota — the same signals Companion binds to its keys.

Change webhook (optional)

Set an outbound URL and the app POSTs to it whenever state changes — handy for pushing to a chat channel or another automation. Clear the field to disable it.

05 Templates & fallback

Put {variables} in a preset's title or description and reuse one preset with different specifics every stream — with a graceful answer for when a value is missing.

Writing variables

Syntax Meaning
{lesson} A fill-in variable, auto-detected from the text. No separate declaration.
{lesson|1} Inline default — 1 is substituted in place when you don't supply a value, keeping the sentence intact.
{{ / }} Escape for a literal { or }.

Variables are allowed only in title and description. A preset with no {…} behaves exactly as it always did and fires immediately.

Per-field fallback

Each of title and description can carry an optional fallback: a complete alternate string used when that field has any unresolved variable. It replaces the whole sentence, not just the gap — so the line still reads naturally. 

title    "أنوار الصحيح الدرس {lesson} - عبد الوهاب"
fallback "أنوار الصحيح - عبد الوهاب"

When lesson is unresolved, the literal الدرس disappears along with it — you get the clean fallback rather than a dangling label.

Resolution order (per field, independent)

  1. Every variable resolves (supplied value → inline default) → render the primary text. (The fill popup prefills inputs with your last-used values, but that only pre-fills the "supplied value" step — the engine itself has no last-used tier.)
  2. Something unresolved and the field has a fallback → render the fallback; those variables report source: "fallback".
  3. Something unresolved and no fallback → reject with MISSING_TEMPLATE_VARS, listing the missing names.

Title and description are evaluated separately — one falling back never drags the other with it.

The fill popup

Selecting a templated preset in the dashboard opens a fill popup instead of firing: one input per detected variable, the inline default / fallback shown as greyed placeholder, last-used values prefilled per preset, and a live read-only preview of the resolved title and description that updates as you type. Submit fires the action; a variable-less preset skips all of this and fires on click.

06 Action endpoints

These change your broadcast. Mounted at both /api/action and /api/dashboard/action — same handler, either path.

Everything returns HTTP 200. Success or failure is in the body (success: true|false), so Companion's HTTP module never has to branch on status codes. On failure the body carries error.code and error.message — bind the message to a variable to flash it on a key.

POST/api/action/preset

Fire a preset. Pass optional vars to fill template variables.

Request

{ "presetId": "gaming_main", "vars": { "lesson": "41" } }

Response

{
  "success": true,
  "resolvedVars": [
    { "name": "lesson", "value": "41", "source": "provided" }
  ]
}

source is one of provided default fallback. A variable in a field that fell back reports fallback. If a required variable is unresolved and its field has no fallback, you get success:false with MISSING_TEMPLATE_VARS.

POST/api/action/update

Ad-hoc override, bypassing presets. All fields except title optional; omitted category / stream binding fall back to app defaults. 

{
  "title": "Custom Input Title",
  "description": "Custom description text",
  "privacyStatus": "public",
  "category": "20",
  "streamBoundId": "abcd-1234-streamkey"
}
POST/api/action/privacy

Flip privacy only, leaving everything else untouched.

{ "status": "private" }  // public | unlisted | private
POST/api/action/undo

Revert the last action. Empty body. Returns NO_UNDO_AVAILABLE when there's nothing to undo.

POST/api/action/refresh

Force an immediate live GET from YouTube and repopulate the cache, bypassing the TTL. Bind this to a manual "Refresh" key for on-demand confirmation right after you change something in YouTube Studio.

Concurrency

A single global busy flag serializes actions. A request that arrives while busy is queued (depth 1) and runs next; a second one that arrives while the slot is full is rejected with BUSY_TRY_AGAIN. This covers the realistic "click, then a catch-up click" without a full queue.

07 Feedback endpoints

Read-only, cache-served, zero YouTube quota. Poll these from Companion as often as you like.

GET/api/feedback/status 
{ "title": "Late Night Gaming Live!", "privacyStatus": "public", "isLive": true }

The core state for on-screen key text and an on-air indicator.

GET/api/feedback/busy 
{ "busy": true }

Drives a "processing" color or blink while an action is in flight.

GET/api/feedback/active-preset 
{
  "activePresetId": "gaming_main",
  "activePreset": {
    "id": "gaming_main",
    "title": "Late Night Gaming — {game}",
    "description": "Come hang out.",
    "privacyStatus": "public",
    "category": "20",
    "streamBoundId": null,
    "titleFallback": null,
    "descriptionFallback": null
  },
  "title": "Late Night Gaming Live!",
  "displayLabel": "Gaming",       // slug → id → "Custom"
  "slugPng": "iVBORw0KGgo…",      // base64 PNG of displayLabel
  "titlePng": "iVBORw0KGgo…",     // base64 PNG of the live title, or null
  "privacyStatus": "public",
  "isLive": true,
  "noTarget": false,
  "busy": false,
  "health": "ok",
  "quotaRemaining": 8760
}
Arabic (and other non-Latin) button text. Companion's bundled fonts have no Arabic glyphs and do no RTL shaping, so an Arabic title bound as plain button text renders as boxes (tofu). Two fields fix that:
  • displayLabel — a short, Latin-safe label: the active preset's slug, or its id when the slug is blank, or "Custom" when no preset is active. Bind it as ordinary button text.
  • slugPng / titlePng — the label and the full live title pre-rendered to a PNG with an Arabic-capable font (shaped + joined), as base64 with no data: prefix. Feed either into a button's image. A common setup binds slugPng and lets a button press toggle to titlePng, since the whole title rarely fits a key. titlePng is null when nothing is on air.
The slug itself may be Arabic — it rides the PNG. The PNGs are rendered only when their text changes and cached, so polling stays cheap. Prefer the raw /slug.png and /title.png endpoints below if your button pulls an image by URL instead of reading base64.

A superset feedback endpoint. It carries the active preset's saved definition in activePreset (its own title, description, privacy, category, stream binding and fallbacks) — so Companion can label or inspect the applied preset without a second call to /api/dashboard/presets. It's null when no preset is active or it was since deleted.

It also folds in the core status, on-air, busy and health signals, so you can bind a whole button — text, color, health lamp, active-preset highlight — from this one poll instead of hitting /status, /busy and /health separately. Note the top-level title is the live broadcast title (what's on air now); activePreset.title is the preset's configured template. The narrow endpoints stay for buttons that need just one field.

GET/api/feedback/slug.png

The button label (slug → id → "Custom") rendered as a 72×72 PNG image — white text with a dark outline so it stays legible over any button colour. For buttons that pull an image by URL rather than reading slugPng base64 out of the state. Always draws something. Returns image/png; changes only when the label changes.

GET/api/feedback/title.png

The full live broadcast title rendered as a 72×72 PNG (Arabic shaped + joined, wrapped, ellipsized if it overflows). Same idea as slug.png, for the whole title. Returns 404 when nothing is on air (no title to draw).

GET/api/feedback/health 
{
  "status": "ok",          // ok | degraded | auth_error
  "authenticated": true,
  "message": null,
  "quotaUsed": 1240, "quotaLimit": 10000, "quotaRemaining": 8760
}
status Meaning Key color
ok Healthy. Green
degraded A transient failure, still retrying. Yellow
auth_error Refresh token dead — needs manual reauth. Red

Health only escalates after HEALTH_FAILURE_THRESHOLD consecutive failures, so a single network blip won't flash your key red. This endpoint is also served unauthenticated at the top level as a liveness check.

GET/api/feedback/stream

Server-Sent Events stream of state changes — an alternative to polling for any custom integration that prefers push. Emits an initial state event on connect, then one per meaningful change.

WS/api/feedback/ws

WebSocket push of the same state changes — Bitfocus Companion's WebSocket module prefers this over SSE. Connect and you get a state frame immediately, then one on every meaningful change (deduped server-side). Frame envelope matches the change webhook: 

{ "event": "state", "state": { "status": { … }, "activePresetId": "gaming_main", "displayLabel": "Gaming", "slugPng": "…", "titlePng": "…", … } }

Also mounted at /api/dashboard/ws (same handler). SSE and 5-second polling remain fully supported — WebSocket is an additional transport, not a replacement. Pick one per client.

Resync. The connection persists across button edits, so a button added after the socket connected would otherwise stay blank until the next change. Two safety nets fix that: state is re-sent on the ~25s heartbeat, and any inbound frame forces an immediate resend. Send any text or hex from the connection (e.g. a Companion "send message" action on button press) and the server replies with the current state at once — no need to reconnect.

09 Error codes

Delivered in the body with HTTP 200. Shape: { success:false, error:{ code, message } }.

Code Meaning
NO_TARGET_FOUND No active broadcast and no persistent container.
INVALID_PRESET presetId not found.
MISSING_TEMPLATE_VARS A required variable is unresolved and its field has no fallback. Lists the names.
BUSY_TRY_AGAIN An action is in flight and the queue slot is full.
NO_UNDO_AVAILABLE Nothing to undo.
YOUTUBE_AUTH_ERROR YouTube rejected the request over token issues.
YOUTUBE_QUOTA_EXCEEDED Daily API quota exhausted.
YOUTUBE_ERROR Any other YouTube API failure.
INVALID_REQUEST Malformed body — the message names the first bad field.

10 Companion · connection

Use Companion's built-in Generic HTTP module. No custom module, no auth header.

  1. In Companion → Connections, add Generic: HTTP Requests.
  2. Set the base URL to http://<APP_IP>:8080.
  3. Leave any authorization header empty — the app is unauthenticated on the LAN.

Every action button below is a POST with a JSON body and Content-Type: application/json. Every feedback binding is a GET polled on the buttons page.

11 Companion · trigger buttons

One key, one preset. Press fires the metadata change on whatever broadcast is current.

A plain preset key

Add a button, give it a HTTP POST action:

URL   http://<APP_IP>:8080/api/dashboard/action/preset
Body  { "presetId": "gaming_main" }

Passing template variables

Fill variables straight from the button using Companion variables or static values: 

Body  { "presetId": "lesson_stream", "vars": { "lesson": "41" } }

If a variable is missing and its field has a fallback, the action still succeeds and the response's resolvedVars tells you the fallback fired. If there's no fallback, you get MISSING_TEMPLATE_VARS — a good case for the redirect flow below, where the operator types the value in a browser first.

12 Companion · state on the keys

Poll the feedback endpoints and bind their fields to key text and colors. All cached — polling costs no YouTube quota.

  1. Add a second HTTP GET connection instance (or per-button feedback) hitting /api/feedback/status, polled every 5 seconds.
  2. Companion parses the JSON into variables. Bind them to buttons:
Endpoint field Bind to
status.title Key text — the on-screen title.
status.isLive Key color — red ring when live, dim when idle.
busy.busy Blink / blue while an action runs.
health.status Green / yellow / red health lamp.
active-preset.activePresetId Highlight the currently applied preset key.
●ON AIR
✓Gaming
◐Working…
◍Health OK

GET /api/feedback/status is served from cache with zero quota — Companion formats and binds these itself; there's no server-side wiring to do.

One poll instead of four. GET /api/feedback/active-preset now returns title, privacyStatus, isLive, busy, health and quotaRemaining alongside activePresetId — bind a whole button from it and skip the separate /status, /busy and /health calls.

Prefer push? Point Companion's WebSocket connection at ws://<APP_IP>:8080/api/feedback/ws for instant updates with no polling interval — the app pushes a state frame on every change. SSE (/api/feedback/stream) and plain 5-second polling stay available if you'd rather not.

Added a button after the connection was already up and it shows no value? The socket doesn't reconnect on a button edit, so it hasn't received a fresh frame yet. Give it a moment — state re-sends on the ~25s heartbeat — or add a "send message" action to the button (any text or hex) to pull the current state instantly. Disabling and re-enabling the connection also forces a fresh frame.

Arabic (non-Latin) titles on a key

Companion's bundled fonts have no Arabic glyphs and do no RTL shaping, so an Arabic title bound as plain key text renders as boxes (tofu). There are two ways around it — pick per how much you need on the key.

Route A — a Latin-safe label (Generic HTTP, works today). Give each preset a short slug in the dashboard and bind the feedback field displayLabel to the key text. It resolves to the preset's slug, or its id when the slug is blank, or "Custom" when no preset is active. Zero extra software. Only limitation: if the slug itself is Arabic it still boxes — that's Route B.

Route B — the real title/slug as an image (custom module). The middleware pre-renders the label and the full title to PNGs with a bundled Arabic font (shaped + joined): fields slugPng / titlePng in the state, and raw images at GET /api/feedback/slug.png and /title.png. Stock Companion can't put a base64 or URL image on a key — its button PNG is a local file only, and Generic HTTP has no image feedback. So this repo ships a small Companion module (companion-module/) that turns those PNGs into two image feedbacks:

  1. Install the module. Easiest (Companion 4+): build a package once and import it — 
    cd companion-module
npm install
npm run package   # runs companion-module-build
    that writes yt-companion-middleware-<version>.tgz; then in Companion go to Modules, click Import module package, and pick that .tgz. No restart needed.
    For development you can instead point Settings → Developer modules path at a folder that contains (or symlinks) companion-module/ after running npm install inside it, then restart Companion — changes reload automatically.
  2. Add the yt-companion-middleware connection; set its base URL to http://<APP_IP>:8080.
  3. On a button, add a feedback → Image: full live title (or Image: button label). The key now shows the Arabic image instead of tofu.
  4. To flip between the short slug and the long title (a title rarely fits a key), make it a two-step button: step 1 uses the slug feedback, step 2 the title feedback — one press toggles. The module also exposes the actions (apply preset, update, privacy, undo, refresh) and the state variables.

The module holds a live WebSocket (instant push) to /api/feedback/ws — no poll interval, updates land the moment state changes, and it auto-reconnects. Template-var presets and opening the dashboard use Companion's built-in Open URL action (bound to $(ytmeta:dashboard_url) or …/fill?preset=…&redirect=…), not the module.

See companion-module/README.md for the full install and packaging notes.

13 Companion · the redirect flow

Companion buttons can't show a popup — so for presets that need the operator to type a value, the button opens a browser page instead, and bounces back to Companion when it's done.

How it flows

Companion Press key Button's action is "open URL", not an HTTP call.
→
Browser Fill page Opens /fill?preset=…&redirect=… with the preset preselected.
→
POST Apply Submit fires /api/dashboard/action/preset.
→
Return Bounce back On success window.location = redirect → back to Companion.

Wire the button

Give the key an Open URL action pointing at the app's fill route, with a redirect back to Companion's own URL (its web control, or a companion:// deep link — any http(s) URL is accepted, no allowlist): 

Open URL
http://<APP_IP>:8080/fill?preset=lesson_stream&redirect=http://<COMPANION_IP>:8000/

What the operator sees

  • The fill popup opens with the preset already selected, last-used values prefilled, and a live preview.
  • On success the page navigates straight to the redirect URL — the operator lands back in Companion.
  • On failure the page stays put and shows the error, so nothing bounces until the action actually went through.
  • A preset with no variables opened this way fires and bounces immediately — no popup at all.

When to use which. Fixed metadata → a plain HTTP POST key (§11). Needs a value typed each time and has no fallback → the redirect flow. Needs a value but has a sensible fallback → either works; the POST key just uses the fallback when you don't pass vars.
