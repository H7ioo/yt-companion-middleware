The button that

A Stream Deck key has no keyboard. The fill flow trades one browser round-trip for the text entry Companion cannot do on its own.

14 Companion · the redirect / fill flow

Companion keys can't prompt for input — and Companion has no "Open URL" action, so a key can't open a browser either. For a preset that needs the operator to type a value, the key instead requests a fill: the middleware surfaces the fill page on whatever screen the operator already has — an open dashboard pops the dialog, and a phone can get a tap-to-open push.

How it flows

Companion Press key Module action Request fill POSTs /api/dashboard/fill-request.
→
Middleware Surface it The request rides the state push to every open dashboard; with ntfy configured, the phone gets a notification too.
→
Operator Fill The fill dialog pops (dashboard) or the notification opens /fill?preset=… (phone).
→
POST Apply Submit fires /api/dashboard/action/preset.

Wire the button

Give the key this module's Request fill (popup on dashboard / phone push) action and pick the preset in its dropdown. That's the whole button — delivery is the middleware's job:

  • Open dashboard (desktop app, or the dashboard open on a phone/tablet over LAN or Tailscale): the fill dialog pops within a second, preset preselected. The first dashboard to claim the request wins — no double popups.
  • Phone push (optional): set an ntfy topic (ntfy.sh, or self-hosted) in the dashboard's Phone push panel and subscribe to it in the ntfy app. The key press then also sends a notification; tapping it opens the fill page in the phone's browser — no dashboard tab needed. Treat the topic name as a secret.
    Same network? Leave Public base URL blank — the link points at this machine's LAN address automatically, even when Companion itself talks to localhost. Fill it in only when the LAN address won't reach the phone: a Tailscale hostname, another subnet, or a reverse proxy.
  • An unclaimed request expires after 60 seconds — a key pressed with nobody watching never pops a stale dialog later.

What the operator sees

  • The fill popup opens with the preset already selected, last-used values prefilled, and a live preview.
  • On success the dialog confirms the apply — the operator closes it and is already back on the dashboard.
  • On failure the dialog stays put and shows the error, so nothing is lost until the action actually went through.

The direct route still exists for anything that can navigate on its own: /fill?preset=<id>[&redirect=<url>] opens the same fill page from a bookmark or a custom integration, and bounces to redirect on success (any http(s) URL, no allowlist).

When to use which. Fixed preset → the Apply preset action or its drag-drop preset button (§11–12). Needs a value typed each time and has no fallback → this Request fill flow. Needs a value but has a sensible fallback → either works; Apply preset just uses the fallback when you leave Template vars blank.

LAN-only. No auth. Keep it off the open internet.
