14 Companion · the redirect / fill flow

Companion keys can't prompt for input — so for a preset that needs the operator to type a value, don't reach for a custom API. Use Companion's built-in Open URL action: the key opens a browser fill page, which POSTs the preset and bounces back to Companion when it's done.

How it flows

Companion Press key Button's action is "open URL", not an HTTP call.
→
Browser Fill page Opens /fill?preset=…&redirect=… with the preset preselected.
→
POST Apply Submit fires /api/dashboard/action/preset.
→
Return Bounce back On success window.location = redirect → back to Companion.

Wire the button

Give the key the built-in Open URL action (not a module action). Point it at the app's fill route — the module already exposes the base URL as $(ytmeta:dashboard_url) — with a redirect back to Companion's own URL (its web control, or a companion:// deep link — any http(s) URL is accepted, no allowlist): 

Open URL
$(ytmeta:dashboard_url)/fill?preset=lesson_stream&redirect=http://<COMPANION_IP>:8000/

To just open the dashboard, drop the path: Open URL → $(ytmeta:dashboard_url).

What the operator sees

  • The fill popup opens with the preset already selected, last-used values prefilled, and a live preview.
  • On success the page navigates straight to the redirect URL — the operator lands back in Companion.
  • On failure the page stays put and shows the error, so nothing bounces until the action actually went through.
  • A preset with no variables opened this way fires and bounces immediately — no popup at all.

When to use which. Fixed preset → the Apply preset action or its drag-drop preset button (§11–12). Needs a value typed each time and has no fallback → this redirect flow. Needs a value but has a sensible fallback → either works; Apply preset just uses the fallback when you leave Template vars blank.
