Every endpoint,

The action, feedback and dashboard routes, and the error codes they answer with. Actions always return HTTP 200 — the outcome is in the body.

06 Action endpoints

These change your broadcast. Served by one handler on two bases, split by caller — /api/action for Companion and /api/dashboard/action for the dashboard. Both supported.

Everything returns HTTP 200. Success or failure is in the body (success: true|false), so Companion's HTTP module never has to branch on status codes. On failure the body carries error.code and error.message — bind the message to a variable to flash it on a key.

POST/api/action/preset

Fire a preset. Pass optional vars to fill template variables.

Request

{ "presetId": "gaming_main", "vars": { "lesson": "41" } }

Response

{
  "success": true,
  "resolvedVars": [
    { "name": "lesson", "value": "41", "source": "provided" }
  ]
}

source is one of provided default fallback. A variable in a field that fell back reports fallback. If a required variable is unresolved and its field has no fallback, you get success:false with MISSING_TEMPLATE_VARS.

POST/api/action/update

Ad-hoc override, bypassing presets. All fields except title optional; omitted category / stream binding fall back to app defaults. 

{
  "title": "Custom Input Title",
  "description": "Custom description text",
  "privacyStatus": "public",
  "category": "20",
  "streamBoundId": "abcd-1234-streamkey"
}
POST/api/action/privacy

Flip privacy only, leaving everything else untouched.

{ "status": "private" }  // public | unlisted | private
POST/api/action/undo

Revert the last action. Empty body. Returns NO_UNDO_AVAILABLE when there's nothing to undo.

POST/api/action/refresh

Force an immediate live GET from YouTube and repopulate the cache, bypassing the TTL. Bind this to a manual "Refresh from YouTube" key for on-demand confirmation right after you change something in YouTube Studio.

Concurrency

A single global busy flag serializes actions. A request that arrives while busy is queued (depth 1) and runs next; a second one that arrives while the slot is full is rejected with BUSY_TRY_AGAIN. This covers the realistic "click, then a catch-up click" without a full queue.

07 Feedback endpoints

Read-only, cache-served, zero YouTube quota. Poll these from Companion as often as you like.

GET/api/feedback/status 
{ "title": "Late Night Gaming Live!", "privacyStatus": "public", "isLive": true }

The core state for on-screen key text and an on-air indicator.

GET/api/feedback/busy 
{ "busy": true }

Drives a "processing" color or blink while an action is in flight.

GET/api/feedback/active-preset 
{
  "activePresetId": "gaming_main",
  "activePreset": {
    "id": "gaming_main",
    "title": "Late Night Gaming — {game}",
    "description": "Come hang out.",
    "privacyStatus": "public",
    "category": "20",
    "streamBoundId": null,
    "titleFallback": null,
    "descriptionFallback": null
  },
  "title": "Late Night Gaming Live!",
  "displayLabel": "Gaming",       // slug → id → "Custom"
  "slugPng": "iVBORw0KGgo…",      // base64 PNG of displayLabel
  "titlePng": "iVBORw0KGgo…",     // base64 PNG of the live title, or null
  "privacyStatus": "public",
  "isLive": true,
  "noTarget": false,
  "busy": false,
  "health": "ok",
  "quotaRemaining": 8760
}
Arabic (and other non-Latin) button text. Companion's bundled fonts have no Arabic glyphs and do no RTL shaping, so an Arabic title bound as plain button text renders as boxes (tofu). Two fields fix that:
  • displayLabel — a short, Latin-safe label: the active preset's slug, or its id when the slug is blank, or "Custom" when no preset is active. Bind it as ordinary button text.
  • slugPng / titlePng — the label and the full live title pre-rendered to a PNG with an Arabic-capable font (shaped + joined), as base64 with no data: prefix. Feed either into a button's image. A common setup binds slugPng and lets a button press toggle to titlePng, since the whole title rarely fits a key. titlePng is null when nothing is on air.
The slug itself may be Arabic — it rides the PNG. The PNGs are rendered only when their text changes and cached, so polling stays cheap. Prefer the raw /slug.png and /title.png endpoints below if your button pulls an image by URL instead of reading base64.

A superset feedback endpoint. It carries the active preset's saved definition in activePreset (its own title, description, privacy, category, stream binding and fallbacks) — so Companion can label or inspect the applied preset without a second call to /api/dashboard/presets. It's null when no preset is active or it was since deleted.

It also folds in the core status, on-air, busy and health signals, so you can bind a whole button — text, color, health lamp, active-preset highlight — from this one poll instead of hitting /status, /busy and /health separately. Note the top-level title is the live broadcast title (what's on air now); activePreset.title is the preset's configured template. The narrow endpoints stay for buttons that need just one field.

GET/api/feedback/slug.png

The button label (slug → id → "Custom") rendered as a 288×288 PNG image — a 72px logical button face drawn supersampled 4× (72 × 4 = 288) so Companion downscales it crisply on larger key surfaces. White text with a dark outline so it stays legible over any button colour. For buttons that pull an image by URL rather than reading slugPng base64 out of the state. Always draws something. Returns image/png; changes only when the label changes.

GET/api/feedback/title.png

The full live broadcast title rendered as a 288×288 PNG (same 72px-supersampled-4× scheme as slug.png; Arabic shaped + joined, wrapped, ellipsized if it overflows). Same idea as slug.png, for the whole title. Returns 404 when nothing is on air (no title to draw).

GET/api/feedback/health 
{
  "status": "ok",          // ok | degraded | offline | auth_error
  "authenticated": true,
  "message": null,
  "quotaUsed": 1240, "quotaLimit": 10000, "quotaRemaining": 8760
}
status Meaning Key color
ok Reaching YouTube normally — actions and background polling are working. Green
degraded A recent call failed and the app is retrying. This usually clears on the next poll. Yellow
offline Can't reach YouTube at the network layer — a firewall, DNS, or dropped internet, not a sign-in problem. Grey
auth_error The saved YouTube sign-in stopped working. No retry will fix it — reconnect to resume actions and status. Red

Health only escalates after HEALTH_FAILURE_THRESHOLD consecutive failures, so a single network blip won't flash your key red. This endpoint is also served unauthenticated at the top level as a liveness check.

GET/api/feedback/stream

Server-Sent Events stream of state changes — an alternative to polling for any custom integration that prefers push. Emits an initial state event on connect, then one per meaningful change.

WS/api/feedback/ws

WebSocket push of the same state changes — Bitfocus Companion's WebSocket module prefers this over SSE. Connect and you get a state frame immediately, then one on every meaningful change (deduped server-side). Frame envelope matches the change webhook: 

{ "event": "state", "state": { "status": { … }, "activePresetId": "gaming_main", "displayLabel": "Gaming", "slugPng": "…", "titlePng": "…", … } }

Also mounted at /api/dashboard/ws (same handler). SSE and 5-second polling remain fully supported — WebSocket is an additional transport, not a replacement. Pick one per client.

Resync. The connection persists across button edits, so a button added after the socket connected would otherwise stay blank until the next change. Two safety nets fix that: state is re-sent on the ~25s heartbeat, and any inbound frame forces an immediate resend. Send any text or hex from the connection (e.g. a Companion "send message" action on button press) and the server replies with the current state at once — no need to reconnect.

08 Dashboard endpoints

What the web UI talks to. You can script these directly too.

Method Path Purpose
GET /api/dashboard/presets List presets.
POST /api/dashboard/presets Create a preset.
PUT /api/dashboard/presets/:id Update a preset.
DEL /api/dashboard/presets/:id Delete a preset.
GET /api/dashboard/presets/export Export all presets (backward-compatible JSON).
POST /api/dashboard/presets/import Import presets from a backup.
GET / PUT /api/dashboard/settings Read / write app-level defaults.
GET /api/dashboard/state Full current state snapshot.
GET /api/dashboard/categories Assignable YouTube categories for the region.
GET /api/dashboard/streams Available stream bindings (keys).
GET / PUT /api/dashboard/webhook Read / set the change webhook URL.
GET /api/dashboard/stream SSE state stream (same as /api/feedback/stream).
WS /api/dashboard/ws WebSocket state push (same as /api/feedback/ws).
POST /api/dashboard/action/* Dashboard base for the action endpoints above — same handler, by caller.

Import / export stays backward compatible. The template fields (titleFallback, descriptionFallback) are optional and nullable — old backups and old presets parse and behave identically.

09 Error codes

Delivered in the body with HTTP 200. Shape: { success:false, error:{ code, message } }.

Code Meaning
NO_TARGET_FOUND No active broadcast and no persistent container.
INVALID_PRESET presetId not found.
MISSING_TEMPLATE_VARS A required variable is unresolved and its field has no fallback. Lists the names.
BUSY_TRY_AGAIN An action is in flight and the queue slot is full.
NO_UNDO_AVAILABLE Nothing to undo.
YOUTUBE_AUTH_ERROR YouTube rejected the request over token issues.
YOUTUBE_QUOTA_EXCEEDED Daily API quota exhausted.
YOUTUBE_ERROR Any other YouTube API failure.
INVALID_REQUEST Malformed body — the message names the first bad field.
LAN-only. No auth. Keep it off the open internet.
